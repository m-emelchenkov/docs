Dash is a working data agent you can clone and run. It answers questions over a SaaS dataset, grounds every answer in curated context, captures its own errors as learnings, and enforces a hard read-only / read-write split. It is every page in this section, shipped.

```bash
git clone https://github.com/agno-agi/dash && cd dash
cp example.env .env   # set OPENAI_API_KEY
docker compose up -d --build
docker exec -it dash-api python scripts/generate_data.py
docker exec -it dash-api python scripts/load_knowledge.py
curl http://localhost:8000/health   # {"status":"ok"}
```

Then ask: *"What's our current MRR?"* or *"Which plan has the highest churn rate?"*

## The team

Dash is a multi-agent system with boundaries enforced by the database, not the prompt.

| Member | Access | Role |
|--------|--------|------|
| **Analyst** | `public`, read-only | Introspects, writes and runs SELECT, answers |
| **Engineer** | `public` read, `dash` read-write | Builds views, records learnings |
| **Leader** | None directly | Routes the request, composes the answer |

## The loop

Retrieve the matching context and prior learnings, generate grounded SQL, answer with a citation, diagnose any error into a learning, and materialize a view when a question recurs. Each error becomes a learning the next run reads and each recurring question becomes a view, so the same errors stop recurring and the `dash` schema accumulates the views your team relies on.

## Adapt it

Dash ships on a synthetic dataset so the first run works. Point it at your warehouse: replace `scripts/generate_data.py` with a loader for your data, then rewrite `knowledge/` for your tables, query patterns, and business rules. The goal is one agent answering one real question correctly, then iterate from real usage.
