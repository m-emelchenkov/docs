An agent that answers is a demo. An agent that does the work is a product. The gap between them is human gates, real tool boundaries, and a scheduler. Agno is built for that gap.

- Actions are risky, so Agno pauses a run on a sensitive tool call and waits for a human.
- Ops work is proactive, so AgentOS has an in-process scheduler. The agent shows up on its own.
- Doing requires tools with limits, so write access is scoped and isolated, not implied by a prompt.
- Work has conventions, so a LearningMachine picks up how your team operates over time.

An agent that does irreversible work needs a gate on it. One decorator pauses the run until a human approves.

```python
from agno.agent import Agent
from agno.models.openai import OpenAIResponses
from agno.tools import tool


@tool(requires_confirmation=True)
def open_pull_request(branch: str, title: str) -> str:
    """Open a PR. Pauses for human approval before it runs."""
    ...


agent = Agent(
    model=OpenAIResponses(id="gpt-5.5"),
    tools=[open_pull_request],
    instructions="Do the work. Sensitive actions pause for approval.",
)

run = agent.run("Open a PR for the auth fix on branch fix/auth")
for req in run.active_requirements:
    if req.needs_confirmation:
        req.confirm()  # or req.reject()
agent.continue_run(run_response=run)
```

The agent proposes the action, the run pauses, a human decides, the run resumes. The boundary holds because it is in the framework, not in the prompt.

## Why Agno fits this shape

| Reason | What it means for an automation system |
|--------|----------------------------------------|
| HITL is native | Any tool can require confirmation; the run pauses and resumes cleanly. |
| Scheduling is in-process | No worker fleet. Cron lives in your `db` and fires from the same service. |
| Tools have boundaries | Read-only vs write agents, sandboxed write surfaces, not prompt promises. |
| It compounds | A shared `LearningMachine` learns your conventions so the work gets more correct. |
