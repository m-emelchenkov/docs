A useful ops agent does not wait to be asked. It posts the morning digest, triages new issues, and syncs state on a cadence. AgentOS has an in-process scheduler, so this needs no Celery, no cron container, no separate worker fleet. Schedules live in your `db` and fire from the same service that serves the agent.

```python
from agno.os import AgentOS

agent_os = AgentOS(
    agents=[agent],
    db=db,
    scheduler=True,
    scheduler_poll_interval=15,  # check for due jobs every N seconds
)
```

The scheduler polls `agno_schedules`, fires due jobs, retries failures, and records every run in `agno_schedule_runs` next to ad-hoc runs.

## Two ways to schedule

| Pattern | When | Lives in |
|---------|------|----------|
| Agent-driven | Users ask the agent to schedule things in chat | The agent's own tool calls |
| Created in code | Schedules that should always exist | A `ScheduleManager` call at startup |

### Agent-driven

Give the agent `SchedulerTools` and it manages cron for you from chat.

```python
from agno.tools.scheduler import SchedulerTools

agent = Agent(
    model=OpenAIResponses(id="gpt-5.5"),
    tools=[
        SchedulerTools(
            db=db,
            default_endpoint="/agents/ops-agent/runs",
            default_method="POST",
            default_timezone="UTC",
        ),
    ],
)
# In Slack: "post a daily digest of open PRs at 9am ET"
```

### Created in code

For jobs that must always exist, create them with `ScheduleManager` at startup. The poller that `scheduler=True` starts picks them up.

```python
from agno.scheduler import ScheduleManager

mgr = ScheduleManager(db)

mgr.create(
    name="daily_digest",
    cron="0 9 * * 1-5",  # weekdays 9am
    endpoint="/workflows/daily-digest/runs",
)
```

`create()` takes `if_exists` (default `"raise"`) to control what happens when a schedule of that name already exists, so a restart neither errors nor duplicates.

## The proactive ops loop

Scheduled work is what turns an agent into operations. Coda ships three jobs by default:

| Job | Cadence | What it does |
|-----|---------|--------------|
| Repo sync | Every 5 minutes | Pulls latest from every configured repo |
| Daily digest | Each morning | Posts what merged, PRs waiting, issues opened, what is stale |
| Issue triage | Daily | Reviews open issues against the actual code, labels them, posts a summary |

The work nobody remembered to do is the work that mattered. A schedule makes it always-on.

## Multi-step scheduled work

A schedule fires one endpoint. When the job is multi-step, point it at a `Workflow`. `Loop` repeats until a condition holds, `Router` branches on input.

```python
from agno.workflow import Step, Workflow

daily_digest = Workflow(
    name="daily-digest",
    steps=[
        Step(name="collect", agent=collector),
        Step(name="summarize", agent=summarizer),
        Step(name="post", agent=slack_poster),
    ],
)
```

Scheduled runs are traced and persisted like any other run, so `agno_schedule_runs` joined with the trace tree tells you exactly what fired and what it did.
